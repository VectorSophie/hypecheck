# Hypecheck v0.2.0: Collision, Drift, and the `compare` / `explain` commands

Four features, shippable independently in build order: **explain â†’ compare â†’
collision â†’ drift**. Each reuses the existing injected `fetchImpl` / `fs` seams.

## Shared groundwork

- **Finding catalog (`src/finding-docs.js`)** â€” a static `FINDING_DOCS` map of
  `id -> { why, verify }` covering every finding id the analyzer emits. Single
  source of truth for what a finding means and how to confirm it yourself.
  Powers `explain`; additive (analyze.js keeps emitting findings unchanged).
- **CLI dispatch (`bin/hypecheck.js`)** â€” refactor the arg parse into a command
  switch: `eval` | `compare` | `explain`. The exit-code contract is unchanged
  for `eval`; new commands define their own (below).

## 1. `explain <finding-id>`

Pure lookup, no network, no scan.

```
hypecheck explain configured-hook
```

Prints the finding's title, **why it matters**, and **how to verify** it (e.g.
"open `hooks/hooks.json` and read the command the hook runs"). Unknown id ->
exit `2` and print the list of known ids. Known id -> exit `0`.

## 2. `compare <a> <b>`

```
hypecheck compare <candidate-a> <candidate-b> [--json] [--no-scan]
```

Runs `evaluateCandidate` on each, then `renderComparison(a, b)` prints both
verdicts, a side-by-side scores table, and the findings unique to each. `--json`
emits `{ a, b }`. Exit code = the worse of the two verdicts (per the existing
INSTALL/TRIAL=0, SKIP/REDUNDANT/DANGEROUS=1 mapping). No engine change.

## 3. Workflow-collision detection

Cross-references the candidate's parsed config against the user's local config â€”
"this collides with what you already run."

- **`local-context.js`**: capture each local hook's *event* (currently the event
  key is discarded at the `Object.values(json.hooks)` loop). Store hooks as
  `{ kind: 'hook', event, name }` (name stays the matcher).
- **`analyzeCollisions(candidateHooks, candidateMcp, localTools, findings)`** (in
  analyze.js, runs for github candidates where manifests exist):
  - candidate hook `event` present in local hooks on the same event ->
    `hook-event-collision` (medium, workflow): "Adds a PostToolUse hook; you
    already run N hook(s) on PostToolUse."
  - candidate MCP server name matches a local MCP server name ->
    `mcp-name-collision` (medium, workflow): "MCP server name `x` already
    configured locally."

Collisions are surfaced as evidence (workflow category). They do not introduce
new verdict rules in v0.2.0 â€” they inform the reader. Slash-command name clashes
are out of scope (we do not enumerate the candidate's commands directory).

## 4. Drift / re-check (rug-pull) â€” opt-in via `--track`

Detects when an already-vetted tool changes its executable surface on a later
version (the rug-pull / post-audit-swap attack class). **Opt-in**: writes nothing
unless `--track` is passed.

- **`src/cache.js`** â€” `readRecord` / `writeRecord` keyed by candidate canonical,
  stored at `~/.hypecheck/evals/<sha256(canonical)>.json`. Cache dir and `fs`
  injectable for tests. Each record: `{ date, version, fingerprint }`.
  - `fingerprint` = stable hash of the candidate's *own public surface*:
    `{ hooks: [{event, command}], mcp: [names], version }`. **No user config, no
    secrets** â€” only what the candidate itself ships.
- **`analyzeDrift(current, prior, findings)`** â€” when `--track` and a prior record
  exists, structurally diff:
  - added hook event(s) or MCP server(s) since last vet -> `drift-detected`
    (**high, security**): "Added a PostToolUse hook since you last checked
    (v1.2 -> v2.0)." This is the textbook rug-pull signal and can push DANGEROUS.
  - version bump with no surface change -> `drift-detected` (**low, maintenance**).
  - first eval (no prior record) -> no finding; write the baseline.
- `--track` writes/updates the record after analysis. Without `--track`, cache is
  neither read nor written. Document the cache location + `--track` in README.

## Verdict / scoring impact

- Collisions (`workflow`): surfaced as evidence; no new verdict rule.
- Drift (`security`, high): flows through existing `securityRisk` / `chooseVerdict`
  like any high security finding, so a rug-pull can reach DANGEROUS.
- `explain` / `compare`: no scoring impact.

## CLI surface summary

```
hypecheck eval <candidate> [--json] [--no-scan] [--scan <path>] [--track]
hypecheck compare <a> <b> [--json] [--no-scan]
hypecheck explain <finding-id>
```

Exit codes unchanged for `eval`; `compare` uses worse-verdict mapping; `explain`
returns `0` (known) / `2` (unknown).

## Non-goals

- npm `--deep` tarball inspection (manifest depth for npm candidates) â€” later.
- `recommended install mode` per verdict â€” later.
- Slash-command name-collision detection â€” needs candidate command enumeration.

## Test plan (node:test, injected fetchImpl + fs + cacheDir)

- `finding-docs` / `explain`: every emitted finding id has a doc entry; unknown id
  path; known id output.
- `compare`: two injected candidates -> side-by-side render; exit = worse verdict.
- `collision`: candidate hook event matches an injected local hook event ->
  `hook-event-collision`; MCP name clash -> `mcp-name-collision`; no clash -> none.
- `local-context`: local hooks now carry `event`.
- `cache`: round-trip read/write with injected dir/fs; missing record -> null.
- `drift`: first eval writes baseline, no finding; changed fingerprint -> high
  `drift-detected`; version-only bump -> low; `--track` off -> no read/write.

## Acceptance

- `hypecheck explain configured-hook` prints why + how-to-verify.
- `hypecheck compare X Y` shows both verdicts side by side.
- A candidate that adds a `PostToolUse` hook on an event you already hook is flagged
  as a collision, naming your local hook count.
- Re-running `eval --track` on a tool that added a hook since the last run reports
  `drift-detected` (high) and can reach DANGEROUS; without `--track`, nothing is
  read or written to `~/.hypecheck/`.
